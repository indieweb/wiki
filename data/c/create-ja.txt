* [[create day|クリエイト・デイ]]

特定の作成UIタイプ：
* [[create event|イベント作成]]
* ...

関連語：''[[creator|クリエイター]]''：自身のサイトのために[[UI]]、[[design|デザイン]]、[[code|コード]]を作成している人。

== モバイル ==
{{main|mobile create|モバイルでの作成}}
モバイルでのノート作成は、様々な理由から特に困難です：
* ネットワークの信頼性が低い（JS / AJAXに依存している場合、ユーザーデータ消失の可能性がある）
* ディスプレイが小さい（入力内容を確認できるスペースが少ない）
* タッチキーボード（物理的なキーボードよりも操作しづらく、正確性に欠ける）

=== Facebook ===
[[Facebook]] は、プロフィールやウォールの「タイムライン」の上部に、以下の3つの明示的な投稿タイプオプションを持つ新規投稿用UIを提供しています：
* 近況 | 写真・動画 | ライフイベント
デフォルトでは「近況」が選択されており、「今なにしてる？」というプロンプトテキストが表示されます。

[[File:Facebook profile create post 2015-09-20.png]]

これは、[[checkin|チェックイン]]やリンク/[[bookmark|ブックマーク]]などがUI上で明示的に区別されていた過去の作成UIから大幅に簡略化されたものです。

==== FB：作成機能の有効化 ====
テキストエリア（デフォルトテキストが表示されている場所）をアクティブにすると、その下に新しいオプションが表示されます：

[[File:Facebook profile create post options 2015-09-20.png]]

アイコンは以下を表します（ホバーするとツールチップが表示されます）：
* 「投稿に写真を追加」
* 「投稿に友達をタグ付け」
* 「気分・アクティビティを追加」
* 「投稿に位置情報を追加」（都市名が自動入力され、削除用の×ボタンが表示される）
* 「投稿の日時を設定」
* 公開範囲の設定（公開、友達、自分のみ（デフォルト）、カスタム、学校、会社など）

==== FB：テキストサイズの自動調整 ====

==== Facebookイベント ====
{{main|create_event#Facebook|イベント作成：Facebook}}
注目すべきことに、上記の [[create|作成]] UIには [[event|イベント]]（人を招待するタイプのもの）を作成する機能が含まれていません。

イベント作成UIは、自身のイベントやカレンダーを表示している時にのみ、文脈的に表示されます。

* `facebook.com/username/events` -> <blockquote><samp>[ + イベントを作成 ] [ ✏️ ]</samp></blockquote> ボタン（鉛筆ボタンは「管理」用ですが、詳細は不明です）
* `https://facebook.com/events` -> 左カラムにある <blockquote><samp>[ + 作成 ]</samp></blockquote> ボタン

「+ [イベントを] 作成」をクリックすると、イベントの詳細を入力するダイアログが表示されます：

[[File:Facebook-create-event-ui.png]]

最初のイベントが作成された後、イベントの [[edit|編集]] UIには既存の写真を選択するオプションが含まれます。詳細は [[edit#Facebook_Events|編集：Facebookイベント]] を参照してください。

イベントの公開/非公開オプション：
* 非公開イベント：「ゲストによる友達の招待を許可」（デフォルト）
* 非公開イベント：主催者のみが友達を招待可能
* 公開イベント：ログインしていない人を含め、誰でも閲覧可能

詳細や、「<span id="Duplicate_Event">イベントの重複</span>」によって発生する「<span id="FB_Something_went_wrong">問題が発生しました</span>」や「<span id="FB_Warning_Slow_Down">警告：操作をスローダウンしてください</span>」といった不可解なエラー、またエラーメッセージが出たにもかかわらず「<span id="FB_Went_Wrong_But_Created_Event">イベントが作成される</span>」問題については、以下を参照してください：
* [[create_event#Facebook|イベント作成：Facebook]]

=== 写真 ===
* このセクションの全内容を独立した [[create photo|写真作成]] ページに移動し、ここには要約を残し、このセクションへのリダイレクトを [[create photo|写真作成]] ページへ向けるように修正することを検討してください。

サーバーに写真をアップロードするプロセスや課題については、[[photo upload|写真アップロード]] を参照してください。

==== UIフロー ====
「体感的な遅延（レイテンシ）」が少ない写真投稿作成体験のためのUIフロー案：

# ユーザーが投稿する写真を選択する。ウェブアプリは即座にバックグラウンドでサーバーへのアップロードを開始する。
# ユーザーが写真をフィルタリングしたりクロップ（切り抜き）したりする。ウェブアプリはフィルターやクロップの指示シーケンスをキャプチャし、ユーザーが編集を「完了」した時に、その指示シーケンスをサーバーに送信する。
# サーバー側でフィルターやクロップを適用する。
# ユーザーが [[person-tags|人物タグ]] やテキストコンテンツ（キャプション）などを追加する。
# ユーザーが「投稿」などをタップする。この時点では送信すべきなのはテキストのみで、他の要素はすべてサーバーに届いているため、投稿は'''即座に'''完了する。

フィルターやクロップの指示シーケンスを純粋に CSS のフィルターおよびクロップ（width, height, object-fit:cover, object-position）プロパティと値として保持し、それらを使用してクライアント側で画像を表示し、サーバーからもその CSS を配信することで、ブラウザにフィルタリングやクロッピングを任せることもできます。これにより、ImageMagick などのサーバーサイドの画像処理ソフトウェアを使用する必要がなくなります。

=== HTMLボックス ===
* 正確なマイクロフォーマットのマークアップに必要なメタデータの量は最小限ですが、多くのオープンソース CMS ソリューションで必要とされるファイルシステムの数がワークフローを複雑にしています。
* プレーンな HTML を受け付け、ユーザーに microformats2 を含めることを求める執筆 UI は、魅力的なユースケースかもしれません。
* どの程度のテンプレート（トップレベルの h* タグなど）を含めるべきかを知る必要があります。
* 投稿タイプ（Like、ブックマークなど）のために URL をパースできる機能があると良いでしょう。

=== オフライン作成・編集、投稿のキューイング ===
ユースケース：iPad 上のオフラインブログ投稿執筆サイト/アプリ。

ネイティブアプリの例：iOS 標準の「メモ」アプリ。十分なコンテンツや構造ができるまで下書きを書き、保存し、拡張し、その後にマークアップを行ったり関連画像を追加したりできます。

関連：
* [[Notion]]
** 批判：90日ごとにログイン（ネットワークアクセス）が必要。[https://twitter.com/NotionHQ/status/1229948659455803394 問題は認識されています。]
* [[local first|ローカルファースト]]
* [[offline first|オフラインファースト]]

ブレインストーミング：
* Service Worker を使用すれば、ウェブサイトで「メモ」アプリのような UI を作り、100% オフラインで編集・保存し、ネットワークに再接続した時に自身のサーバーと「同期」させることが可能なはずです。
* それだけでも非常に有用なので、既に存在しているはずです（「メモ」アプリのようなクリーンな UX を模倣する CSS はそれほど難しくないはずです）。ここで言っているのは、近年の「メモ」アプリに追加された派手なオブジェクトではなく、「シンプル」なテキストノートのことです。
** {{snarfed}} による以前の考察： https://snarfed.org/posting-to-the-indieweb-from-your-phone
** JS/Localstorage を使った「メモ」アプリ構築のYouTubeチュートリアル： https://www.youtube.com/watch?v=01YKQmia2Jw
*** JS/Localstorage 「メモ」アプリの最小構成の HTML+CSS+JS： https://github.com/dcode-youtube/notes-app-javascript-localstorage/tree/main
* 「公開（Publish）」を独立した別のアクションとし、それを実行するとノートが「凍結」されて公開キューに入れられるようになると素晴らしいでしょう。もちろん、公開キューから取り除いて編集を続けるために、自動的に凍結を解除できるライフサイクルも備えている必要があります。
* ネットワークに再接続した時に、キューに入ったノートを Micropub で自動公開できます。
* 拡張案：公開の「タイムウィンドウ」を設定できるようにする。例えば、自分の活動時間帯（昼間）にのみ自動公開したい場合など。さらに、Buffer のように特定の公開時間を予約できる機能（[[scheduled post|予約投稿]]）を追加します。
