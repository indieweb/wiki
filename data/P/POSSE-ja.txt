'''<dfn>POSSE</dfn>''' は、'''Publish (on your) Own Site, [[:Category:syndication|Syndicate]] Elsewhere'''（自分のサイトで公開し、他所に配信する）の略称です。これは、コンテンツをまず自分のサイトに投稿し、その後、コピーを公開したりサードパーティ（[[social media]] siloなど）にリンクを共有したりする手法を指します。その際、閲覧者があなたのコンテンツと直接やり取りできる経路を提供するために、[[original post link]]（元の投稿へのリンク）を添えます。

== なぜやるのか ==

'''友人に、彼らの好きな方法であなたの投稿を読んでもらいましょう。''' POSSEを使えば、友人はあなたのコンテンツを読むために、彼らが普段使っているもの（[[Instagram]]、[[Tumblr]]、[[Twitter]]、[[Neocities]]などのソーシャルメディア [[silo]]）を使い続けることができます。

'''「いつか」ではなく「今」、友人と連絡を取り合いましょう。''' POSSEは、将来的に友人とつながる可能性よりも、現在の友人と今つながり続けることを重視しています。 

'''連邦化（Federation）よりも友人が大切です。''' 建築的な理想よりも人々にとって重要な人間関係に焦点を当てることで、人間的な観点から見れば、POSSEは[[federation]]よりも重要です。さらに、連邦化のアプローチがまずPOSSEの手法を取り入れれば、より採用されやすくなり（誰もが友人と連絡を取り合いたいと思っているため）、その結果、連邦化された未来により急速に近づくことができます。

'''POSSEはブログの域を超えています。''' これは、IndieWebのムーブメントが単なる「全員が自分のサイトでブログを書く」ことや、「全員がお気に入りのソーシャルソフトウェアをインストールして実行する」といった[[monoculture]]（単一文化）的な解決策とは[[different]]（異なる）理由であり、その方法の核となる部分です。

=== 全般的な理由 ===

<span id="Advantages">POSSEは、以下の理由から、堅牢で好ましい配信モデルであると考えられています</span>。

* '''サードパーティへの依存を減らす。''' 自分のサイトに直接投稿することで、投稿のためにサードパーティのサービスに依存しなくて済みます。自分のサイトにアクセスできれば、コンテンツを公開できます。対照的にPESOSでは、[https://twitter.com/colinjohnston/status/772533151960707072 サードパーティのサイトがダウンしているとき]、コンテンツを追加することができません。
* '''所有権。''' 自分のサイトに「最初に」投稿することで、中間にサードパーティのサービス（サイロ）の利用規約（TOS）を挟むことなく、自分にまで遡れる直接的な所有権の連鎖を構築できます（これは[[PESOS]]の脆弱性です）。
* '''コンテンツの正規URLを所有する。''' コンテンツへの正規URL（Canonical URL）を自分のドメイン上に持てます。
* '''コピーからオリジナルを引用できる。''' コンテンツを最初に自分のサイトに投稿する（つまりパーマリンクを作成する）ことで、サードパーティのサービスに投稿するコピーから、自分のサイトのオリジナルへリンクしたり引用したりできます（[[syndication_formats]] および [[Twitter#POSSE_Notes_to_Twitter|TwitterへのPOSSEノート]] を参照）。
* '''より優れた検索。''' 自分のドメイン上の公開コンテンツを（任意の検索エンジンを使って）検索する方が、投稿の検索を[[silos]]だけに頼るよりも効果的です（例えば、Twitterは一時期、検索結果に最近のツイートしか表示しませんでした。[[Facebook]]の検索結果はいまだに非常に貧弱です）。
* [[backfeed]]（バックフィード）を使用して、他のサービスからの反応を取り込む（逆配信する）ことができます。
* コンテンツの正規コピーを自分のサイトに保存しながら、他のサービスのソーシャルレイヤーや集計機能を活用できます。
* ...

=== なぜオリジナルへリンクするのか ===
一般的な[[POSSE]]の実践では、[[permashortlink]]を使用して、POSSEコピーからオリジナルへリンクを貼ります。その理由は以下の通りです。

* '''オリジナルコンテンツの発見。''' サードパーティのサービスに投稿されたオリジナルへのパーマリンクにより、それらのサービス上のコピーからオリジナルのコンテンツを[[discovery]]（発見）することが可能になります。
* '''投稿をコピーするスパマーを逆手に取る。''' スパマー（例: @sin3rss）があなたのPOSSEコピーを無意識にコピーして転載したとき、彼らはオリジナルへのリンクも一緒にコピーします。その結果、より多くの人があなたのオリジナルの投稿を見つけ、閲覧するための導線となります。ある種の「[http://tantek.com/2011/009/t3/value-searching-finding-past-tweets-self-hosting 2011-01-09 インターネット合気道]」です。
* '''オリジナル投稿のランキング向上。''' あなたのPOSSEコピーが他者によってコピーされ、他所へ（手動リツイートやRSSボットなどを介して）転載された場合、それらのコピーがオリジナルへリンクしていれば、検索エンジンはそれらのリンクを辿ってオリジナルを特定し、ランクを高く設定します。

== 方法 ==
=== 実装方法 ===
このセクションは、POSSEを実装するウェブ開発者向けです。
==== 全般 ====
一般的に、コンテンツ投稿ソフトウェアが何かを投稿するとき、選択した[[silo]]の宛先にもコピーを投稿し、そこからオリジナルへ戻る[[original post link]]（例: [[permashortlink]] や [[permashortcitation]]）を含めるべきです。

その詳細は宛先によって異なります。以下のサイロ別のセクションを参照してください。

コピーをサイロに投稿した後は、以下のことを行うべきです：
* オリジナル投稿の '''[[posts-elsewhere]]''' セクションから、配信されたコピーへリンクを貼る。

==== ユーザーインターフェース ====
最高のユーザーインターフェース（UI）とは、自動的で、信頼でき、目に見えないものです。常に期待通りに、予測可能な形でPOSSEを実行できるのであれば、明示的なUIは必要ありません。

===== プレビュー =====
予測可能性を高め、安心感を与える一つの方法は、公開前に配信先にどのように表示されるかを（配信先の制限内で）プレビューとして表示することです。

（スクリーンショットが必要）

==== Twitter ====
{{main|Twitter#POSSE_to_Twitter}}
[[Twitter]]はおそらく最も人気のあるPOSSEの宛先であり、始めるのに適した場所です。

ノート（ツイート）をTwitterに直接投稿するのではなく、自分のサイトに投稿してTwitterにPOSSEし始めることができれば、データの所有権を握るための大きな一歩を踏み出したことになります。

詳細：
* APIアクセス - 永続的なAPIトークンにより、[https://developer.twitter.com/en/docs/tweets/post-and-engage/api-reference/post-statuses-update.html 新しいツイートの投稿]はスムーズに機能し、戻り値には投稿されたURLが含まれます。
** 2022年11月現在、Twitterは、Twitterのルールやポリシーに違反する可能性があるという理由で、POSSE/バックフィードに使用されるアプリケーションの新しいAPIアクセスを拒否しています。 — {{barnabywalters}}
* 非常に完成度の高いWebアクションエンドポイントをサポートしているため、半手動の投稿を簡単に実装できます。
** これらのエンドポイントは何ですか？2022年現在もそうですか？ — {{barnabywalters}}

ノートとアーティクル（ブログ投稿）の両方をTwitterにPOSSEする方法の詳細については、[[Twitter#POSSE_to_Twitter|TwitterへのPOSSE]]を参照してください。

==== 公開フロー ====
POSSEのコンテンツ投稿フローを実装する方法は、少なくとも2つあります：

===== クライアント → サイト → サイロ =====
* ユーザーが公開用クライアントを使用してコンテンツを作成する
** オプション：クライアントがサードパーティサービスを認識している場合、それらを選択するUIや、サービスごとのカスタマイズ機能を提供する
* コンテンツが完成したら、ユーザーは自分のサーバーに公開する（オプション：どのサードパーティサービスに送るか、およびそのカスタマイズ内容などのメタデータを付与する）
** オプション：クライアントはサーバーの状態を把握した上でパーマリンクを生成し、そのパーマリンクに対して公開を行うことができる
* サーバーがコンテンツを公開し、必要に応じてパーマリンクと要約（および/またはサードパーティサービスに適したカスタマイズコンテンツ）を生成する
* サーバーが、パーマリンク付きのコピーをサードパーティサービスに投稿する

利点：
* ユーザーはインターネット上で自分のサイトという1つの場所とだけやり取りすればよい
* 配信（Syndication）をサーバー側で完全に自動化できる
欠点：
* 特になし？

===== クライアント → サイト および サイロ =====
* ユーザーが公開用クライアントを使用してコンテンツを作成する
* コンテンツが完成したら、ユーザーは自分のサーバーに公開する
* クライアントは、プッシュしたばかりのコンテンツのURLをサーバーに問い合わせる
* 公開用クライアントは、以下を選択するためのインターフェースをユーザーに提示する：
** どのサードパーティサービスに公開するか
** サービスに公開する正確な内容（生成されたコンテンツに基づく要約が事前入力されている）
* ユーザーがサービスを選択し、フォームを送信する
* 公開用クライアントが、コンテンツの要約をサードパーティサービスに投稿する

利点：
* サードパーティサービスへのコンテンツコピーのタイミングや編集について、ユーザーがより細かく制御できる
欠点：
* 配信のたびに手動の手順が必要になる
* クライアントからサードパーティサービスへの直接的な接続性に依存する（モバイル接続が不安定な状況や、クライアントがドメイン検閲のあるインターネットアクセスを使用して公開している場合に問題となる）

== 他のアプローチ ==
=== COPE ===
<abbr>COPE</abbr> は Create Once, Publish Everywhere（1度作って、どこでも公開）の略。これには最初の「1度公開（Publish Once）」というステップが明示的に欠けており、コンテンツを様々な宛先に複製することに主眼が置かれています。

自分が「所有（Own）」するサイトでの「1度公開」ステップがなく、したがって [[original post permalink]] も欠けているため、COPE 戦略は、自分のコンテンツを読んだり見たりするための正規の場所に人々を引き寄せることに失敗します。その結果、他人のサイト上で（おそらく）バラバラの聴衆を増やすだけに終わります。

記事：
* 2009-10-13 [https://web.archive.org/web/20140517203502/http://www.programmableweb.com/news/cope-create-once-publish-everywhere/2009/10/13 COPE: Create Once, Publish Everywhere] （Daniel Jacobson 著、NPR アプリケーション開発ディレクター）。（オリジナルの <nowiki>https://www.programmableweb.com/news/cope-create-once-publish-everywhere/2009/10/13</nowiki> は、2022年の <nowiki>programmableweb.com</nowiki> の[[site-death-ja|サイト消滅]]によりオフライン）
* 2019-10-28 [https://www.smashingmagazine.com/2019/10/create-once-publish-everywhere-wordpress/ Create Once, Publish Everywhere With WordPress] （Leonardo Losovitz 著、Smashing Magazine）
* 2019 WordCamp Taipei トーク： [https://wordpress.tv/2020/06/16/leonardo-losoviz-create-once-publish-everywhere-with-wordpress-2/ Create Once, Publish Everywhere] （WordPress.tv のビデオ）

=== POSE ===
{{main|POSE}}
[[POSE]]（Publish Once Syndicate Everywhere）は、POSSE のより広範な前身。特定の [[silo]] で 1 度公開し、そこから他のサイロへ配信することも含まれていました。

=== PESOS ===
{{main|PESOS}}

似ているが逆のアプローチが [[PESOS]]。コンテンツをまずサードパーティのサービスに投稿し、それを個人のサイトにコピー/配信する手法です。

コンテンツの正確なコピーが個人サイトとサードパーティサービスの両方に投稿されている場合、POSSE か PESOS かを（おそらく存在しない、1秒未満の正確な公開日時を比較しない限り）見分ける方法はありません。配信されたコピーに、公開されたオリジナルへ戻るパーマ(ショート)リンクが含まれていれば、そのサイトが POSSE をサポートしていることを証明できます。

=== PESETAS ===
{{main|PESETAS}}
[[PESETAS]] は PESOS に似ているが、（個人サイトを一切介さずに）すべてを特定のサイロにコピー/配信する手法。

例えば、ほとんどのサイロは [[Twitter]] へのクロスポストをサポートしているため、すべてを Twitter アカウントに接続し、常にコピーを保持するためにそこへ（自動）クロスポストすることができます。

例：[[Tumblr]] には Twitter へクロスポストするための UI がある。[http://webapps.stackexchange.com/questions/15315/customize-the-message-tumblr-sends-to-twitter Webapps StackExchange の投稿] にドキュメントと UI のスクリーンショットがあります。

ただし、Tumblr は十分に確立されており、より多様なコンテンツを扱え、より多くのテキストを許可し、Twitter のようにリンクをラップせずに URL へ直接リンクできるため、PESETAS の宛先としてはより適しています。

== ブレインストーミング ==
=== CRUD ===
これまで（2013-222）のところ、POSSE は自身のサイトでのコンテンツの作成（'''C'''reation / 公開）を他のサイトに配信することのみを指してきました。このモデルは非常に成功しており、おそらくそれで十分かもしれません。

しかし、POSSE 用の完全な CRUD プロトコルの潜在的な有用性を探る価値はあります。

=== Create（作成） ===
Create は POSSE のデフォルトです。自身のサイトでコンテンツを作成し、その Create を他のサイトに POSSE します。これについては上記、および [[silo]] ページのサイロ別の詳細で説明されています。

=== Read（読み取り） ===
動詞としての Read を POSSE に適用すると興味深いものになります。

最低限、配信されたコピーからの将来的な読み取りの可能性に備えて、コンテンツの配信先コピーへのリンクを保存しておくことは有用です。

参照：
* [[u-syndication]]：コンテンツの配信先コピーへのリンクをマークアップする方法
* [[syndication-link-use-cases]]：なぜそれを行うのか

配信先 POSSE コピーからの読み取りの実際の直接的な用途：
* オリジナル投稿に対する、POSSE コピー周辺のアクティビティの逆配信 / [[backfeed]]（バックフィード）：
** [[comments]]/返信
** [[favorites]]/いいね
** [[reposts]]（リツイート）

さらに、POSSE コピーへの [[u-syndication]] リンクを保持しておくことで、後述する Update や Delete アクションを実行するために、そのコピーを削除することが可能になります。

=== Update（更新） ===
もし POSSE の宛先が更新/編集を許可しているなら、投稿を編集した際、その更新を配信先の POSSE コピーにも伝播させることができます。
* 例：[[Facebook]] は投稿のテキスト（テキスト内のリンクを含む）や人物タグの編集を許可しているが、写真投稿の画像自体の変更は許可していない。

[[Twitter]] のように宛先が更新/編集を許可していない場合でも、POSSE ツイートを削除して再投稿することで、仮想的に POSSE 更新を実行することが可能です。すなわち：

Twitter への POSSE 更新のみを検討する場合：
* まだ誰も返信していない場合（そうしないと Twitter 上のツリー形式の会話を壊してしまうため）
* 変更内容が Twitter 上の切り詰められたコピーに表示される場合（つまり、変更箇所が 140文字（実質 120文字）の地平線より先であれば、Twitter コピーを書き換える意味がない）
* 非常に短い時間枠（おそらく 2〜5分以内）である場合。そうでないと、Twitter であなたを読んでいる人々にとって、更新が重複として見えてしまうため。

これらの懸念はすべて、Twitter であなたのツイートを読んでいる友人への体験に関するものです。これこそが、そもそも Twitter へ POSSE する手間をかける唯一の（設計上の）理由であるべきです。

詳細は、サイロ別の POSSE セクションを参照してください：
* [[Facebook]]：[[POSSE to Facebook]] （[[to-do]]：テキスト編集はOKだが画像編集は不可、写真投稿は POSSE 更新をシミュレートするために削除/再投稿が必要、という詳細が必要）
* [[Flickr]]： (UI は写真投稿の画像の手動更新をサポートしているが、API で利用可能か？もしそうなら、Flickr への POSSE 更新（写真投稿の画像を含む）をサポートするために、[[Bridgy]] Publish の機能リクエストを GitHub に起票すること)
* [[Twitter#POSSE_to_Twitter|Twitter: POSSE to Twitter]] （[[to-do]]：上記の削除/再投稿戦略をそこにコピーする）
* ……

=== Delete（削除） ===
削除をPOSSEで行うのは、特に削除をクライアントへ伝播させるサービスに対しては、比較的簡単に見えます。

例：Twitterのノートはいつでも削除可能です。

Update（更新）と同様に、以下の点を考慮してください：
* すでにPOSSEコピーに返信（あるいは「いいね」やリツイートのようなアクティビティ）がある場合、会話のスレッド（および他人の「いいね」やリツイート）を維持するために、あえて削除せずに残しておくことを検討してください。

しかし、自分のサイトとPOSSEコピー（例：Twitter）の両方からコンテンツを本当に削除したい場合は、そのまま実行してください。

おそらく、投稿の削除用UIにおいて、削除を実行する前にPOSSEコピーにアクティビティ（返信、いいね、リツイート）があったかどうかを確認できるようにするのは良い機会です。実装案としては、UIがユーザーにこれらのアクティビティの有無を通知し、サービスごとに削除のリクエストを再確認させる方法が考えられます。


==== IndieWebの実装例 ====

* {{grantcodes}} は、2018-10-10時点でTwitterでのPOSSE削除をサポートしています。自身のサイトの投稿が非公開または削除されたかどうかを確認し、いいね、リポスト、ノートに対して適切なAPIリクエストを送信します。

== FAQ ==
=== 検索エンジンと重複コンテンツの心配 ===
Q：検索エンジンが、重複していると思われる投稿にペナルティを与えることを心配する必要はありますか？

A：そのために、POSSEコピーは常にオリジナルへリンクを貼る「べき（SHOULD）」なのです。そうすることで、検索エンジンはコピーが単なるコピーであることを推測できます。理想的には、サイロ上のPOSSEコピーは [[rel-canonical]] を使用してオリジナルへリンクすべきですが、明示的な rel-canonical がなくても、オリジナルへの明示的なリンクは、それがオリジナルであるという強力なヒントになります。

これは、PESOSに対するPOSSEの利点でもあります。PESOSの場合、何がオリジナルで何がコピーかを判別する方法がないため、それらは単なる重複に見えてしまいます。

=== POSSE投稿の発見とバックリンク ===
Q：Brid.gy は、明示的なリンクがない場合でも、[[posse-post-discovery]] を使用して配信された投稿とオリジナルの関係を見つけることができます。これは、配信されたコピーに [[backlink]] を追加するのをやめてもいいということですか？

A：[[backlink]] なしのPOSSEは「最後の手段」と考えられており、いくつかのコストが伴います。詳細は [[posse-post-discovery#Tradeoffs]] を参照してください。

=== POSSEとWebmention送信のどちらを優先すべきか ===
結論から言えば、'''まず [[POSSE]] を行い、その後に [[webmentions]] を送信してください'''。

詳細と理由については、[[Webmention-faq#POSSE_or_Send_Webmentions_First|Webmention FAQ: POSSEとWebmention送信のどちらを優先すべきか]] を参照してください。
